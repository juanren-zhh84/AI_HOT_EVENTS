# GitHub 热点项目自动化监控服务需求文档

版本：v0.1  
日期：2026-06-01  
技术语言：Python  
建议服务形态：Python 后端服务 + 定时任务 + 邮件投递

## 1. 项目背景

本项目用于自动化监控 GitHub 项目的星级增长数据，按天筛选近期热度上升的开源项目，并归集项目功能介绍、技术选型、仓库指标等信息，最终以邮件日报的形式定时发送给订阅者。

服务需要支持长期运行、可配置监控范围、可追踪每日增长趋势，并能在 GitHub API 限流、网络异常、邮件发送失败等情况下稳定恢复。

## 2. 建设目标

1. 自动采集 GitHub 项目的基础信息、星标总量、语言、Topic、README 等数据。
2. 按固定频率记录星标快照，形成日级、周级增长趋势。
3. 每日根据热度规则筛选热点项目。
4. 自动生成项目功能介绍与技术选型摘要。
5. 按预设时间通过邮件发送热点项目日报。
6. 提供管理接口，用于配置监控源、触发任务、查询结果、维护订阅者。

## 3. 使用角色

| 角色 | 说明 | 主要操作 |
| --- | --- | --- |
| 系统管理员 | 维护服务配置与数据源 | 配置 GitHub Token、邮件服务器、调度规则、监控源 |
| 内容订阅者 | 接收每日热点项目邮件 | 接收日报、退订或调整订阅偏好 |
| 运维人员 | 监控系统运行状态 | 查看任务状态、失败日志、GitHub API 限流情况 |

## 4. 功能范围

### 4.1 监控源管理

系统应支持以下监控源：

| 类型 | 示例 | 说明 |
| --- | --- | --- |
| 指定仓库 | `openai/openai-python` | 直接监控固定仓库 |
| GitHub Search 查询 | `topic:ai stars:>1000` | 按关键词、Topic、语言、星标数筛选候选项目 |
| Topic | `machine-learning` | 周期性发现某个 Topic 下的新项目 |
| 组织或用户 | `huggingface/*` | 监控组织或用户下的公开仓库 |

监控源应支持启用、禁用、优先级、采集频率、过滤规则。

### 4.2 GitHub 数据采集

系统应采集以下数据：

| 数据类型 | 字段示例 |
| --- | --- |
| 仓库基础信息 | owner、repo、full_name、description、homepage、html_url、created_at、updated_at、pushed_at |
| 仓库指标 | stargazers_count、forks_count、watchers_count、open_issues_count |
| 分类信息 | language、topics、license |
| 星标快照 | snapshot_at、stars、star_delta_1h、star_delta_24h、star_delta_7d |
| README 信息 | README 原文、摘要、更新时间 |
| 技术选型 | 主要语言、框架、包管理器、运行环境、部署相关文件 |

采集逻辑要求：

1. 对每个仓库定期记录星标总数快照。
2. 对新发现仓库立即采集完整基础信息。
3. 使用 GitHub API 分页能力获取列表类数据。
4. 使用 GitHub API 限流信息控制请求速率。
5. 对 403、404、429、5xx 等异常进行分类重试或跳过。
6. 对已删除、私有化、转移地址的仓库记录状态变化。

### 4.3 热点项目筛选

每日定时任务应基于最近 24 小时数据筛选热点项目。默认筛选条件如下：

| 条件 | 默认值 | 说明 |
| --- | --- | --- |
| 最低总星标数 | 100 | 排除低质量噪声项目 |
| 24 小时新增星标 | >= 20 | 关注短期增长 |
| 7 日新增星标 | >= 50 | 关注持续增长 |
| 最近提交时间 | 180 天内 | 排除长期不维护项目 |
| 仓库状态 | 非 archived、非 disabled | 只保留可访问项目 |

推荐热度评分公式：

```text
hot_score =
  stars_delta_24h * 0.45 +
  stars_delta_7d * 0.25 +
  growth_rate_24h * 100 * 0.15 +
  activity_score * 0.10 +
  freshness_score * 0.05
```

评分字段说明：

| 字段 | 说明 |
| --- | --- |
| `stars_delta_24h` | 最近 24 小时新增星标 |
| `stars_delta_7d` | 最近 7 天新增星标 |
| `growth_rate_24h` | 24 小时新增星标 / 采集前总星标 |
| `activity_score` | 最近提交、Issue 活跃度、Release 活跃度综合分 |
| `freshness_score` | 新项目或近期首次爆发项目加权 |

### 4.4 功能介绍归集

系统应为热点项目生成结构化介绍：

| 字段 | 说明 |
| --- | --- |
| 一句话简介 | 基于仓库 description 与 README 首页内容生成 |
| 主要功能 | 提取 README 中的功能点、使用场景、核心能力 |
| 适用人群 | 开发者、数据科学家、运维、企业用户等 |
| 使用门槛 | 安装复杂度、依赖环境、是否需要 API Key 或模型权重 |
| 项目亮点 | 相比同类项目的差异点 |

摘要生成优先级：

1. 优先使用仓库 description、topics、README 标题与前几段内容。
2. README 过长时先进行内容清洗和分段截取。
3. MVP 阶段可使用规则提取；后续可接入大模型生成更自然的摘要。
4. 对无法解析 README 的项目，保留原始 description 并标记 `summary_status=partial`。

### 4.5 技术选型识别

系统应基于以下信息识别技术栈：

| 来源 | 可识别内容 |
| --- | --- |
| GitHub languages | 主语言、语言占比 |
| `package.json` | Node.js、React、Vue、Next.js、Express 等 |
| `pyproject.toml`、`requirements.txt` | Python、FastAPI、Django、Flask、PyTorch 等 |
| `go.mod` | Go 模块与依赖 |
| `Cargo.toml` | Rust 项目依赖 |
| `pom.xml`、`build.gradle` | Java、Spring、Gradle、Maven |
| `Dockerfile`、`docker-compose.yml` | 容器化与服务依赖 |
| `.github/workflows` | CI/CD、测试、发布流程 |

输出结果应包含：

```json
{
  "primary_language": "Python",
  "frameworks": ["FastAPI", "SQLAlchemy"],
  "runtime": ["Python 3.12"],
  "database": ["PostgreSQL"],
  "deployment": ["Docker"],
  "confidence": 0.86
}
```

### 4.6 邮件日报

系统应每日定时生成并发送邮件，默认发送时间为北京时间 09:00。

邮件内容应包括：

1. 邮件标题：`GitHub 热点项目日报 - YYYY-MM-DD`
2. 今日热点项目总数。
3. Top N 项目列表，默认 Top 20。
4. 每个项目展示：
   - 项目名称与 GitHub 链接
   - 一句话简介
   - 今日新增星标、总星标、7 日新增星标
   - 主要语言与技术栈
   - 项目亮点
5. 数据采集时间与统计时间窗口。
6. 退订或订阅管理入口。

邮件发送要求：

1. 支持 SMTP 配置。
2. 支持 HTML 和纯文本两种格式。
3. 发送失败时自动重试。
4. 每封邮件记录发送状态、失败原因、重试次数。
5. 支持测试邮件发送。

### 4.7 管理接口

系统应提供 REST API，用于：

1. 健康检查。
2. 管理监控仓库和监控源。
3. 手动触发采集任务。
4. 查询采集任务状态。
5. 查询星标快照。
6. 查询每日热点项目。
7. 生成或发送邮件日报。
8. 管理订阅者。
9. 查看 GitHub API 限流状态。

接口建议使用 FastAPI 实现，并通过 OpenAPI 自动生成在线接口文档。

## 5. 非功能需求

### 5.1 技术栈建议

| 模块 | 推荐技术 |
| --- | --- |
| Web API | Python 3.12+、FastAPI、Pydantic |
| HTTP 客户端 | httpx |
| 调度任务 | APScheduler；规模变大后可切 Celery + Redis |
| 数据库 | PostgreSQL；本地开发可用 SQLite |
| ORM | SQLAlchemy |
| 邮件模板 | Jinja2 |
| 邮件发送 | smtplib 或第三方邮件服务 SDK |
| 配置管理 | pydantic-settings、`.env` |
| 部署 | Docker、Docker Compose |
| 日志监控 | structlog 或标准 logging |

### 5.2 性能要求

1. 支持至少 10,000 个仓库的日级快照存储。
2. 单次日报生成耗时不超过 5 分钟。
3. API 查询 P95 响应时间不超过 500ms。
4. 采集任务应支持并发，但必须遵守 GitHub API 限流。

### 5.3 稳定性要求

1. 定时任务重复执行时必须幂等。
2. 采集任务中断后可从最近状态恢复。
3. 邮件重复发送应有防重机制，避免同一日报重复投递。
4. 关键任务失败需要记录日志，并暴露可查询状态。

### 5.4 安全要求

1. GitHub Token、SMTP 密码等敏感配置只能从环境变量或密钥服务读取。
2. 管理 API 需要鉴权，MVP 可使用 Bearer Token。
3. 日志中不得打印 Token、SMTP 密码、订阅者完整敏感信息。
4. 订阅者邮箱需要支持禁用和退订。

## 8. 任务调度

| 任务 | 默认频率 | 说明 |
| --- | --- | --- |
| 仓库发现任务 | 每 6 小时 | 从 Search、Topic、组织源发现候选仓库 |
| 星标快照任务 | 每 1 小时 | 记录仓库星标变化 |
| 项目信息补全任务 | 每日 02:00 | 拉取 README、语言、依赖文件并生成摘要 |
| 热点项目计算任务 | 每日 08:30 | 生成当日热点榜单 |
| 邮件日报发送任务 | 每日 09:00 | 发送热点日报 |
| 失败任务重试 | 每 15 分钟 | 重试可恢复失败 |

## 9. 验收标准

MVP 版本需要满足：

1. 可以添加至少 50 个 GitHub 仓库并完成星标快照采集。
2. 可以基于 24 小时与 7 日增长计算热点项目排名。
3. 可以为热点项目生成基础功能介绍和技术栈字段。
4. 可以配置订阅者并发送 HTML 邮件日报。
5. 可以通过 API 查询仓库、快照、热点榜单、任务状态。
6. GitHub API 限流或短时失败时，服务不会崩溃，并能记录错误。
7. 管理 API 具备基础 Bearer Token 鉴权。

## 10. 里程碑建议

| 阶段 | 目标 | 交付物 |
| --- | --- | --- |
| M1 | 项目骨架与数据库 | FastAPI 服务、数据表、配置管理 |
| M2 | GitHub 采集 | 仓库采集、星标快照、限流处理 |
| M3 | 热点计算 | 热度评分、榜单查询、历史趋势 |
| M4 | 摘要与技术栈 | README 摘要、依赖识别、技术选型输出 |
| M5 | 邮件日报 | 模板渲染、SMTP 发送、订阅者管理 |
| M6 | 部署与监控 | Docker、日志、健康检查、任务状态页 |

## 11. 主要风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| GitHub API 限流 | 采集不完整 | 使用 Token、分页、缓存、条件请求、任务分片 |
| README 格式差异大 | 摘要质量不稳定 | 规则提取 + 可选大模型总结 |
| 仓库爆发速度快 | 小时级采样可能遗漏细节 | 热点候选项目提高采集频率 |
| 邮件被判垃圾邮件 | 到达率下降 | 配置 SPF/DKIM、控制发送频率、提供退订 |
| 数据规模增长 | 查询变慢 | 建索引、按日期分区、归档历史快照 |
